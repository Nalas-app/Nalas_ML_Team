## 2. Deployment Strategy — Separate Python Service

### 2.1 Why a Separate Service?

| Consideration | Decision | Rationale |
|--------------|----------|-----------|
| **Language** | Python (separate from Node.js backend) | ML libraries (scikit-learn, XGBoost, pandas) are Python-native |
| **Framework** | FastAPI | Async, high-performance, auto-generated OpenAPI docs |
| **Deployment** | Separate process on same/different server | Independent scaling, independent restarts |
| **Communication** | REST API (HTTP/JSON) | Simple, well-understood, debuggable |
| **Model Loading** | At service startup (in-memory) | Fast inference, no disk I/O per request |

### 2.2 Service Configuration

```yaml
# ml_service/config.yaml
service:
  host: "0.0.0.0"
  port: 8001
  workers: 2
  timeout: 30
  log_level: "info"

model:
  store_path: "./model/"
  active_version: "v0.1.0"
  fallback_enabled: true

database:
  host: "localhost"
  port: 5432
  name: "nalas_db"
  
monitoring:
  metrics_enabled: true
  log_predictions: true
  log_path: "./logs/predictions/"
```

## 3. Model Storage Strategy

### 3.1 Phase 1: Filesystem Storage (Current)

```
project_root/
└── model/
    ├── active_model.json          # {"active_version": "v0.1.0", "method": "rule_based"}
    ├── v0.1.0/
    │   └── rule_config.json       # Business rule parameters
    └── v1.0.0/                    # (Future) First ML model
        ├── cost_model.pkl
        ├── feature_config.json
        ├── metrics.json
        └── model_card.md
```

**Pros:** Simple, no additional infrastructure, sufficient for early stages.  
**Cons:** No versioning history, no access control, manual backups needed.

### 3.2 Phase 2+: Object Storage (Future Migration)

```
cloud_storage/
└── nalas-ml-models/
    ├── v0.1.0/
    ├── v1.0.0/
    ├── v1.1.0/
    └── ...
```

**Migration Trigger:** When the system has 3+ model versions and/or moves to cloud deployment.

## 4. Deployment Procedures

### 4.1 Initial Deployment (First Time)

```bash
# 1. Clone project
git clone <repository-url>
cd nalas_ml_costing

# 2. Create virtual environment
python -m venv .venv
source .venv/bin/activate   # Linux/Mac
.venv\Scripts\activate      # Windows

# 3. Install dependencies
pip install -r requirements.txt

# 4. Verify model file exists
ls model/cost_model.pkl

# 5. Start ML service
cd ml_service
uvicorn main:app --host 0.0.0.0 --port 8001 --workers 2

# 6. Verify health
curl http://localhost:8001/
# Expected: {"message": "Nalas ML Costing Engine is running"}
```

### 4.2 Model Update Deployment

```bash
# 1. Place new model in versioned directory
mkdir -p model/v1.0.0/
cp new_cost_model.pkl model/v1.0.0/cost_model.pkl
cp feature_config.json model/v1.0.0/
cp metrics.json model/v1.0.0/

# 2. Update active model pointer
echo '{"active_version": "v1.0.0", "method": "ml_model"}' > model/active_model.json

# 3. Restart ML service (graceful)
kill -SIGTERM <pid>
uvicorn main:app --host 0.0.0.0 --port 8001 --workers 2

# 4. Verify new model is active
curl http://localhost:8001/ml/model-info
# Expected: {"version": "v1.0.0", "method": "ml_model", ...}

# 5. Run smoke test
curl -X POST http://localhost:8001/ml/predict-cost \
  -H "Content-Type: application/json" \
  -d '{"menuItemId": "test-uuid", "quantity": 100}'
```

### 4.3 A/B Testing Deployment

```
┌──────────────────────────────────────────────────┐
│              A/B TESTING FLOW                    │
│                                                  │
│  Request ───▶ Traffic Router                     │
│                    │                             │
│           ┌────────┴────────┐                    │
│           │                 │                    │
│        90% traffic      10% traffic              │
│           │                 │                    │
│     ┌─────▼─────┐    ┌─────▼─────┐              │
│     │ Current   │    │ New       │              │
│     │ Model     │    │ Model     │              │
│     │ v0.1.0    │    │ v1.0.0    │              │
│     └─────┬─────┘    └─────┬─────┘              │
│           │                 │                    │
│           └────────┬────────┘                    │
│                    │                             │
│              Log both predictions                │
│              Compare after N requests            │
│                                                  │
│  Rollout: 10% → 50% → 100% (if metrics pass)   │
│                                                  │
└──────────────────────────────────────────────────┘
```

**A/B Testing Phases:**

| Phase | Traffic Split | Duration | Success Criteria |
|-------|--------------|----------|-----------------|
| Shadow Mode | 100% old + parallel new (no user impact) | 1 week | New model predictions logged, no errors |
| Canary | 10% new, 90% old | 1 week | New model MAPE ≤ old model MAPE |
| Partial Rollout | 50% new, 50% old | 1 week | Consistent performance |
| Full Rollout | 100% new | Permanent | Monitor for 2 weeks post-rollout |

## 5. Environment Management

| Environment | Purpose | Port | Model Version |
|-------------|---------|------|---------------|
| **Development** | Local development & testing | 8001 | Latest or experimental |
| **Staging** | Pre-production validation | 8001 | Candidate version |
| **Production** | Live predictions | 8001 | Stable, validated version |

## 6. Scaling Considerations

### 6.1 Current Scale (Phase 1)
- Expected load: < 100 predictions/day
- Single service instance is sufficient
- No load balancing needed

### 6.2 Future Scale (Phase 3+)
| Load Tier | Predictions/Day | Infrastructure |
|-----------|----------------|---------------|
| Low | < 100 | Single instance, 2 workers |
| Medium | 100 - 1,000 | Single instance, 4 workers |
| High | 1,000 - 10,000 | 2 instances behind load balancer |
| Very High | 10,000+ | Kubernetes with auto-scaling |
