## 2. Fallback Scenarios

### 2.1 ML Model Unavailable → Rule-Based Costing

**Trigger Conditions:**
- Model file not found on disk
- Model fails to load (corrupt file, version mismatch)
- Model inference throws an exception
- Model returns invalid output (negative values, NaN)

**Fallback Behavior:**
```python
def predict_cost(request):
    try:
        # Attempt ML model prediction
        result = ml_model.predict(features)
        
        if not is_valid_prediction(result):
            raise ValueError("Invalid ML prediction output")
        
        return format_response(result, method="ml_model")
    
    except Exception as e:
        # Fallback to rule-based
        logger.warning(f"ML model failed, using rule-based fallback: {e}")
        result = rule_based_calculator.calculate(request)
        
        return format_response(result, method="rule_based", 
                               fallback_reason=str(e))
```

**Response includes:** `"method": "rule_based"` to indicate fallback was used.

### 2.2 Database Connection Fails → Cached Prices

**Trigger Conditions:**
- Database connection timeout (> 5 seconds)
- Database connection refused
- Query execution error

**Fallback Behavior:**
```python
class PriceService:
    def __init__(self):
        self._cache = {}           # In-memory price cache
        self._cache_timestamp = {} # Last update time per ingredient
    
    def get_ingredient_price(self, ingredient_id):
        try:
            # Try database first
            price = self.db.query_price(ingredient_id)
            
            # Update cache on success
            self._cache[ingredient_id] = price
            self._cache_timestamp[ingredient_id] = datetime.now()
            
            return price
        
        except DatabaseConnectionError:
            # Fallback to cache
            if ingredient_id in self._cache:
                cache_age = datetime.now() - self._cache_timestamp[ingredient_id]
                
                if cache_age.days > 7:
                    logger.warning(f"Using stale cached price for {ingredient_id} "
                                   f"(cached {cache_age.days} days ago)")
                
                return self._cache[ingredient_id]
            
            # No cache available
            raise PriceUnavailableError(f"No price available for {ingredient_id}")
```

**Cache Invalidation Rules:**
| Condition | Action |
|-----------|--------|
| Successful DB query | Update cache entry |
| Price updated in DB | Invalidate & refresh on next access |
| Cache entry > 24 hours | Mark as "stale" but still usable |
| Cache entry > 7 days | Log warning when used |
| Cache entry > 30 days | Reject and return error |

### 2.3 Prediction Fails → Error with Actionable Message

**Trigger Conditions:**
- All fallback levels exhausted
- Required data completely unavailable
- Input validation failure

**Error Response Format:**
```json
{
    "error": {
        "code": "ML_005",
        "message": "Cost prediction temporarily unavailable",
        "details": {
            "reason": "Model inference failed and rule-based calculation could not complete",
            "suggestion": "Please try again in a few minutes or enter cost manually",
            "support_reference": "ERR-20260328-001"
        }
    }
}
```

## 3. Fallback Decision Matrix

| ML Model | Database | Cache | Result |
|----------|----------|-------|--------|
| ✅ Available | ✅ Available | ✅ Fresh | ML prediction (normal path) |
| ❌ Failed | ✅ Available | ✅ Fresh | Rule-based calculation |
| ✅ Available | ❌ Failed | ✅ Fresh | ML prediction with cached data |
| ❌ Failed | ❌ Failed | ✅ Fresh | Rule-based with cached prices |
| ❌ Failed | ❌ Failed | ⚠️ Stale | Rule-based with stale prices + warning flag |
| ❌ Failed | ❌ Failed | ❌ Empty | Error response with actionable message |

## 4. Rollback Procedures

### 4.1 Model Rollback (New Model → Previous Version)

**When to Rollback:**
- MAPE exceeds 20% for 3 consecutive days after new model deployment
- API error rate exceeds 5% after deployment
- Latency P99 exceeds 1 second consistently
- Critical bug discovered in new model

**Rollback Steps:**
```
┌──────────────────────────────────────────────────────────┐
│                MODEL ROLLBACK PROCEDURE                  │
│                                                          │
│  Step 1: Identify issue                                  │
│    └─▶ Check monitoring dashboard for anomalies          │
│    └─▶ Confirm issue is model-related (not infra)        │
│                                                          │
│  Step 2: Initiate rollback                               │
│    └─▶ Update active_model.json to previous version      │
│        echo '{"active_version": "v0.1.0"}' >             │
│             model/active_model.json                      │
│                                                          │
│  Step 3: Restart ML service                              │
│    └─▶ Graceful restart:                                 │
│        kill -SIGTERM <pid>                                │
│        uvicorn main:app --host 0.0.0.0 --port 8001      │
│                                                          │
│  Step 4: Verify rollback                                 │
│    └─▶ GET /ml/model-info → confirm old version active   │
│    └─▶ Send test prediction → verify response            │
│    └─▶ Check error rate returning to normal               │
│                                                          │
│  Step 5: Post-mortem                                     │
│    └─▶ Document what went wrong                          │
│    └─▶ Root cause analysis                               │
│    └─▶ Fix and re-validate before re-deploying           │
│                                                          │
│  TARGET: Complete rollback within 15 minutes              │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 4.2 Rollback Checklist

- [ ] Issue confirmed as model-related (not infrastructure)
- [ ] Previous model version identified and available on disk
- [ ] `active_model.json` updated to point to previous version
- [ ] ML service restarted gracefully
- [ ] Health endpoint returning 200
- [ ] Model info endpoint showing correct version
- [ ] Test prediction returning valid results
- [ ] Monitoring confirms error rate dropping
- [ ] Backend team notified of rollback
- [ ] Incident documented

### 4.3 Service Rollback (Code Changes)

**When to Rollback:**
- Bug in API code causing errors
- Feature engineering logic producing wrong results
- Configuration error

**Procedure:**
```bash
# 1. Identify the last known good commit
git log --oneline -5

# 2. Revert to last good commit
git revert <bad-commit-hash>
# OR
git checkout <last-good-hash> -- ml_service/

# 3. Restart service
kill -SIGTERM <pid>
uvicorn main:app --host 0.0.0.0 --port 8001 --workers 2

# 4. Verify service is healthy
curl http://localhost:8001/
curl http://localhost:8001/ml/model-info
```

## 5. Monitoring During Fallback

When any fallback is active, the following additional monitoring applies:

| Metric | Threshold | Action |
|--------|-----------|--------|
| Fallback usage rate | > 0% | Log and track |
| Fallback usage rate | > 20% for 1 hour | Warning alert → investigate ML model |
| Fallback usage rate | > 50% for 1 hour | Critical alert → immediate investigation |
| Fallback usage rate | 100% for 30 min | Critical alert → ML service likely down |
| Cache staleness (avg) | > 24 hours | Warning alert → check DB connectivity |
| Error response rate | > 5% | Critical → all fallbacks failing |

## 6. Recovery Procedures

### 6.1 Recovery from ML Model Failure
```
1. Check model file integrity: ls -la model/v*.*.*/
2. Try reloading model: restart ML service
3. If model corrupt: restore from backup or rollback to previous version
4. If persistent: investigate model training pipeline
```

### 6.2 Recovery from Database Failure
```
1. Check database connectivity: psql -h localhost -U nalas -d nalas_db
2. Check database logs for errors
3. Restart database service if needed
4. Verify ML service can connect: check /ml/health endpoint
5. Manually refresh cache: trigger cache warmup
```

### 6.3 Recovery from Complete Service Failure
```
1. Check service process: ps aux | grep uvicorn
2. Check port availability: netstat -tlnp | grep 8001
3. Check logs: tail -100 logs/ml_service.log
4. Restart service: uvicorn main:app --host 0.0.0.0 --port 8001
5. Verify health: curl http://localhost:8001/
6. Notify backend team that ML service is back online
```
