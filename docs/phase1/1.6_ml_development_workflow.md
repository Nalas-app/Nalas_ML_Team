### 1.2 Phase Details

#### Phase A: Experimentation (Local Notebooks)

**Purpose:** Explore data, test hypotheses, prototype features.

**Activities:**
- Data exploration and visualization (`notebooks/` directory)
- Feature engineering experiments
- Quick prototyping of different algorithms
- Statistical analysis of data patterns

**Rules:**
- All notebook work stays in `notebooks/` directory
- Notebooks are for exploration only, NOT for production code
- Document findings in markdown cells
- Save key insights before moving to training phase

**Output:** Feature list, algorithm shortlist, data insights report

---

#### Phase B: Model Training (Training Scripts)

**Purpose:** Formal model training with reproducible pipeline.

**Activities:**
- Write training scripts (not notebooks) for production models
- Implement feature engineering pipeline (reusable code)
- Hyperparameter tuning (GridSearchCV / RandomizedSearchCV)
- Cross-validation (5-fold stratified)

**Rules:**
- Training code lives in `ml_service/training/` directory
- All randomness must be seeded (`random_state=42`)
- Training data split must be reproducible
- Log all hyperparameters and metrics

**Output:** Trained model file, training metrics, feature config

---

#### Phase C: Evaluation (Test Set Validation)

**Purpose:** Validate model quality before deployment.

**Activities:**
- Evaluate on held-out test set (never seen during training)
- Calculate MAPE, MAE, RMSE, R²
- Residual analysis (no systematic patterns)
- Feature importance analysis
- Edge case testing
- Error analysis (which items have highest error?)

**Rules:**
- Test set is NEVER used during training or tuning
- All metrics documented in `metrics.json`
- Feature importance must make business sense
- Edge cases explicitly tested and documented

**Output:** Evaluation report, model card, go/no-go recommendation

---

#### Phase D: Production Deployment

**Purpose:** Deploy validated model to production API.

**Activities:**
- Export model with versioned directory structure
- Update API to load new model
- A/B testing (shadow mode → canary → full rollout)
- Monitor for 48 hours post-deployment
- Documentation updates

**Rules:**
- Model must pass all evaluation criteria before deployment
- A/B testing required for any new model version
- Rollback plan must be tested before deploying
- Monitor actively for first 48 hours

**Output:** Deployed model, updated API, monitoring active

## 2. Code Review Process

### 2.1 What Requires Code Review

| Change Type | Review Required | Reviewer |
|-------------|----------------|----------|
| ML model code (training, inference) | ✅ Yes | ML team member + Lead |
| Feature engineering logic | ✅ Yes | ML team member |
| API endpoint changes | ✅ Yes | ML team member + Backend team |
| Business rule changes | ✅ Yes | ML Lead + Business stakeholder |
| Configuration changes | ✅ Yes | ML team member |
| Documentation updates | ⚠️ Soft review | Any team member |
| Notebook experiments | ❌ No (but share findings) | N/A |

### 2.2 Code Review Checklist

#### For ML Model Code:
- [ ] Code is clean, well-documented, and follows project conventions
- [ ] No hardcoded values (use config files)
- [ ] Feature engineering is consistent between training and inference
- [ ] Random seeds are set for reproducibility
- [ ] No data leakage (future data doesn't leak into training)
- [ ] Input validation present
- [ ] Error handling implemented
- [ ] Unit tests written and passing
- [ ] Performance metrics documented

#### For API Code:
- [ ] Request validation using Pydantic schemas
- [ ] Error responses follow error code registry
- [ ] Prediction logging implemented
- [ ] Fallback logic tested
- [ ] Response format matches API contract
- [ ] No sensitive data in logs or responses
- [ ] Latency is acceptable (< 500ms P99)

## 4. Code Review Guidelines

### 4.1 ML-Specific Review Points

| Review Area | What to Check | Why It Matters |
|-------------|--------------|----------------|
| **Data Leakage** | Does any feature use information from the target or future? | Inflates metrics artificially; model fails in production |
| **Train-Test Contamination** | Is test data strictly separate from training? | Gives falsely optimistic metrics |
| **Feature-Target Alignment** | Are features computed at prediction time (not after)? | Some features may be unavailable at inference time |
| **Reproducibility** | Are random seeds set? Is data split deterministic? | Others must be able to reproduce results |
| **Scalability** | Does the code handle the expected data volume? | Processing 1000 orders is different from 100 |
| **Numerical Stability** | Division by zero? Log of zero? Overflow? | Edge cases cause silent failures |

### 4.2 Common ML Code Smells

| Smell | Example | Fix |
|-------|---------|-----|
| **Leaking target** | Using `actual_cost` as a feature | Remove; use only pre-prediction features |
| **Using future data** | Price data from after the order date | Use price snapshot at prediction time |
| **Overfitting** | Train MAPE 2%, Test MAPE 20% | Regularization, simpler model, more data |
| **Magic numbers** | `if quantity > 500: discount = 0.1` | Use config file |
| **No error handling** | `model.predict(x)` with no try/except | Wrap in try/except with fallback |
| **No validation** | Accepting any input without checks | Add Pydantic validation |

## 7. Git Workflow

### 7.1 Branch Strategy

| Branch | Purpose | Merges To |
|--------|---------|-----------|
| `main` | Production-ready code | — |
| `develop` | Integration branch | `main` (after review) |
| `feature/xxx` | New feature development | `develop` |
| `model/xxx` | Model training & experiments | `develop` |
| `hotfix/xxx` | Urgent production fixes | `main` + `develop` |

### 7.2 Commit Message Convention

```
type(scope): description

Types: feat, fix, model, data, docs, test, refactor, config
Scope: api, training, features, monitoring, etc.

Examples:
feat(api): add seasonal adjustment to cost prediction
model(v1.0): train Random Forest with 100 orders dataset
fix(api): handle missing ingredient price gracefully
docs(phase1): add monitoring metrics specification
data(prices): update ingredient prices for March 2026
test(api): add edge case tests for extreme quantities
```
