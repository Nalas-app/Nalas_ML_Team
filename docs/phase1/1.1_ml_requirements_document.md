## 1. Problem Statement

**Prediction Objective:**  
Given a menu item, quantity, and optional event details (event date, guest count), predict a **complete cost breakdown** including ingredient cost, labor cost, overhead cost, and total cost with profit margin.

**Formal Definition:**  
```
f(menu_item, quantity, event_date?, guest_count?) → {
    ingredient_cost,
    labor_cost,
    overhead_cost,
    total_cost,
    confidence_score,
    model_version,
    prediction_method
}
```

## 2. Performance Requirements

| Metric | Target | Measurement |
|--------|--------|-------------|
| **Prediction Accuracy (MAPE)** | < 10% | Rolling 30-day window on completed orders |
| **Response Latency (P99)** | < 500ms | 99th percentile end-to-end prediction time |
| **Service Uptime** | > 99% | Monthly availability |
| **Confidence Threshold** | ≥ 60% | Predictions below 60% flagged for manual review |

## 3. Business Constraints

All predictions **must** adhere to the following hard constraints:

### 3.1 Cost Positivity
- All cost components (ingredient, labor, overhead, total) **must be positive** (> 0).
- A prediction returning zero or negative cost is treated as a system error.

### 3.2 Mandatory Cost Components

| Component | Calculation Rule | Basis |
|-----------|-----------------|-------|
| **Ingredient Cost** | Sum of (ingredient_qty × unit_price) for all recipe ingredients | Recipe + Price Master |
| **Labor Cost** | 15% of ingredient cost | Fixed multiplier |
| **Overhead Cost** | 10% of ingredient cost | Fixed multiplier |
| **Subtotal** | Ingredient + Labor + Overhead | Sum |
| **Profit Margin** | 10% of subtotal | Fixed multiplier |
| **Total Cost** | Subtotal + Profit Margin | Final price |

**Example Calculation:**
```
Ingredient Cost  = ₹10,000
Labor Cost       = ₹10,000 × 0.15 = ₹1,500
Overhead Cost    = ₹10,000 × 0.10 = ₹1,000
Subtotal         = ₹10,000 + ₹1,500 + ₹1,000 = ₹12,500
Profit Margin    = ₹12,500 × 0.10 = ₹1,250
Total Cost       = ₹12,500 + ₹1,250 = ₹13,750
```

### 3.3 Seasonal Pricing Adjustments

| Season | Period | Adjustment |
|--------|--------|------------|
| **Wedding Season** | Oct – Feb | +10% to +20% on ingredient costs |
| **Festival Season** | Mar – Apr (Pongal, Navratri, etc.) | +5% to +15% on ingredient costs |
| **Off-Season** | May – Sep | No adjustment (baseline) |

> Seasonal adjustments are applied **before** labor/overhead multipliers.

### 3.4 Minimum Cost Thresholds

| Item Category | Minimum Cost per Serving (₹) |
|---------------|------------------------------|
| Vegetarian Main Course | 50 |
| Non-Vegetarian Main Course | 100 |
| Rice / Biryani | 60 |
| Dessert / Sweet | 40 |
| Starter / Appetizer | 45 |
| Beverage | 20 |

> If a prediction falls below the category minimum, it is clamped to the minimum and flagged for review.

### 3.5 Maximum Variance Rule
- Any prediction that deviates **more than ±30%** from the historical average cost for the same item + quantity range must be flagged for manual review.
- This serves as an anomaly detection guardrail.

## 4. Data Requirements

### 4.1 Currently Available Data
| Data Source | Description | Records | Quality |
|-------------|-------------|---------|---------|
| NRC MENU.xlsx | 31 recipe sheets, inventory list, price sheet | ~96 items | Partial (many recipes missing quantities) |
| NRC Kitchen_Outsource-Inn.xls | 130 ingredient prices, 6+ complete recipe cost breakdowns | ~130 items | Good (prices are usable) |
| dummy_catering_data.csv | Synthetic training data (500 records) | 500 | Synthetic only |

### 4.2 Data Gaps (Critical)
| Missing Data | Impact | Mitigation |
|-------------|--------|------------|
| Historical order data | Cannot validate predictions against actuals | Use rule-based approach initially; collect data over time |
| Labor time per dish | Cannot compute real labor cost | Use fixed 15% multiplier |
| Overhead allocation | Cannot compute real overhead | Use fixed 10% multiplier |
| Seasonal demand patterns | Cannot model seasonal pricing accurately | Apply static seasonal adjustments |
| Complete recipe quantities | Many recipes have ingredient lists but no quantities | Work with kitchen team to fill gaps |

### 4.3 Data Collection Targets
| Milestone | Target | Timeline |
|-----------|--------|----------|
| Complete recipe quantities for all menu items | 100% recipes with quantities | Month 1-2 |
| First 50 completed orders logged | Basic pattern identification | Month 2-3 |
| 100+ completed orders with actual costs | Train first ML model | Month 4-6 |
| 500+ completed orders | Train advanced ML model | Month 8-12 |

## 5. Success Criteria

### 5.1 Production Readiness Criteria

| Criterion | Threshold | Verification Method |
|-----------|-----------|-------------------|
| Prediction Accuracy (MAPE) | < 10% | Backtest on hold-out set + rolling 30-day production monitoring |
| Prediction Latency (P99) | < 500ms | Load testing with 100 concurrent requests |
| Service Uptime | > 99% | Monthly availability reports |
| Confidence Coverage | > 80% of predictions have confidence ≥ 60% | Distribution analysis of confidence scores |
| Zero Critical Bugs | 0 P0/P1 bugs open | Bug tracker review before go-live |

### 5.2 Model Retraining Criteria

| Trigger | Action |
|---------|--------|
| MAPE exceeds 15% for 7 consecutive days | Investigate and retrain model |
| Ingredient prices change by > 20% (batch) | Retrain with updated price data |
| 100 new orders collected since last training | Schedule retraining |
| Quarterly (minimum) | Scheduled retraining regardless of triggers |

### 5.3 Model Versioning Convention
```
v{MAJOR}.{MINOR}.{PATCH}

MAJOR = New algorithm or major architecture change (e.g., v1 → v2)
MINOR = Retraining with new data or feature additions (e.g., v1.0 → v1.1)
PATCH = Bug fixes, config changes, no model change (e.g., v1.1.0 → v1.1.1)
```

**Current Version:** `v0.1.0` (Rule-based MVP)

## 7. Assumptions & Dependencies

### Assumptions
1. Recipe data provided by NRC is accurate and represents current menu.
2. Business multipliers (labor 15%, overhead 10%, profit 10%) are approved by management.
3. Seasonal adjustments are uniform across all menu categories.
4. The system will initially operate in **rule-based mode** until sufficient historical data is collected.
