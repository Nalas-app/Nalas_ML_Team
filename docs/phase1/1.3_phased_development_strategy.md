## 1. Strategy Overview

Given the **current data constraints** (limited historical orders, incomplete recipe quantities, no actual cost tracking), we adopt a **4-phase approach** that delivers value immediately while building toward a fully ML-powered costing engine.

│                                                                     │
│  Phase 1          Phase 2           Phase 3           Phase 4       │

│  MVP              COLLECTION        MODEL             MODEL         │
│                                                                     │
│  ████████████     ░░░░░░░░░░░░     ░░░░░░░░░░░░     ░░░░░░░░░░░░  │
│  ▲ CURRENT                                                          │

## 2. Phase 1: Rule-Based MVP (Immediate Implementation)

### 2.1 Objective
Deploy a **deterministic cost calculation engine** that uses existing recipe and ingredient price data to generate reliable cost quotations — **no ML model required**.

### 2.2 Available Data

| Data | Source | Status |
|------|--------|--------|
| Ingredient prices (130 items) | NRC Kitchen_Outsource-Inn.xls | ✅ Available |
| Recipe compositions (31 dishes) | NRC MENU.xlsx | ⚠️ Partial (some missing quantities) |
| Complete cost breakdowns (6 dishes) | NRC Kitchen_Outsource-Inn.xls | ✅ Available |
| Inventory categories | NRC MENU.xlsx | ✅ Available |

### 2.3 Implementation Plan

```
┌─────────────────────────────────────────────────────────┐
│              PHASE 1 IMPLEMENTATION                     │
│                                                         │
│  1. Build Ingredient Price Master                       │
│     └─▶ Extract from NRC Kitchen file                   │
│     └─▶ Normalize units (kg, g, ml, L, piece)           │
│     └─▶ Store in database                               │
│                                                         │
│  2. Build Recipe Database                               │
│     └─▶ Extract available recipes with quantities       │
│     └─▶ Map ingredients to price master                 │
│     └─▶ Define base serving size per recipe             │
│                                                         │
│  3. Implement Cost Calculator                           │
│     └─▶ ingredient_cost = Σ(qty × price × scale)       │
│     └─▶ labor_cost = ingredient_cost × 0.15             │
│     └─▶ overhead_cost = ingredient_cost × 0.10          │
│     └─▶ subtotal = ingredient + labor + overhead        │
│     └─▶ profit = subtotal × 0.10                        │
│     └─▶ total = subtotal + profit                       │
│                                                         │
│  4. Apply Business Rules                                │
│     └─▶ Seasonal adjustments                            │
│     └─▶ Bulk discounts                                  │
│     └─▶ Minimum cost thresholds                         │
│                                                         │
│  5. Deploy FastAPI Endpoint                             │
│     └─▶ POST /ml/predict-cost                           │
│     └─▶ Input validation                                │
│     └─▶ Error handling                                  │
│                                                         │
│  6. Enable Prediction Logging                           │
│     └─▶ Log every prediction request + response         │
│     └─▶ This becomes future training data               │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 2.4 Phase 1 Deliverables

| Deliverable | Description | Status |
|-------------|-------------|--------|
| Ingredient price master (database table) | Cleaned, normalized prices | To Do |
| Recipe database (structured) | Recipes with ingredients + quantities | To Do |
| Rule-based cost calculator | Python module with business logic | To Do |
| FastAPI prediction endpoint | REST API for cost predictions | Partially Done (basic endpoint exists) |
| Prediction logging | Every prediction logged with full details | To Do |
| Unit tests | Tests for cost calculation logic | To Do |

### 2.5 Phase 1 Success Criteria

- [ ] API returns valid cost breakdown for all menu items with complete recipes
- [ ] Cost components match business rules (15% labor, 10% overhead, 10% profit)
- [ ] Seasonal adjustments applied correctly
- [ ] Response latency < 200ms (P99)
- [ ] All predictions logged for future ML training
- [ ] Error handling returns appropriate error codes

## 3. Phase 2: Data Collection Period

### 3.1 Objective
Collect sufficient **real-world order data** to enable ML model training. This phase runs in parallel with Phase 1 production usage.

### 3.2 Data Collection Requirements

| Data Point | Source | Collection Method |
|-----------|--------|-------------------|
| **Order details** (menu items, quantities, guest count) | Backend order system | Auto-captured on order creation |
| **Event details** (date, type, location) | Backend order system | Auto-captured on order creation |
| **Predicted cost** (at quotation time) | ML Service logs | Auto-logged per prediction |
| **Actual cost** (after order completion) | Backend + kitchen team | Manual entry initially, automated later |
| **Ingredient prices at time of order** | Price database snapshot | Auto-captured with each prediction |
| **Customer feedback** (accepted/rejected quotation) | Backend | Auto-captured on quotation status change |

### 3.3 Data Collection Milestones

```
Month 1        Month 2        Month 3        Month 4
──────────────────────────────────────────────────────
█░░░░░░░░░     ██░░░░░░░░     ████░░░░░░     ██████████
~10 orders     ~30 orders     ~60 orders     ~100 orders
                                              ▲
                                              │
                                         ML TRAINING
                                         THRESHOLD
```

| Milestone | Target | Significance |
|-----------|--------|-------------|
| 10 completed orders | Month 1 | Initial data validation, schema verification |
| 30 completed orders | Month 2 | Preliminary pattern identification |
| 50 completed orders | Month 3 | Basic statistical analysis possible |
| **100 completed orders** | **Month 4** | **Minimum viable training dataset** |
| 200 completed orders | Month 6 | Reliable model training |
| 500 completed orders | Month 9-12 | Advanced model training |

### 3.4 Data Quality Monitoring

| Check | Frequency | Action if Fails |
|-------|-----------|----------------|
| All required fields populated | Per order | Alert → manual data entry reminder |
| Actual cost recorded within 7 days of order completion | Daily | Reminder notification to kitchen team |
| Predicted vs actual cost logged | Per order | Auto-check in prediction logging |
| No duplicate orders | Daily | Deduplication script |
| Ingredient prices current (< 30 days) | Weekly | Alert to update prices |

### 3.5 Phase 2 Deliverables

| Deliverable | Description |
|-------------|-------------|
| Data collection pipeline | Automated order + cost data capture |
| Prediction vs actual tracking | Dashboard/report comparing predictions to actuals |
| Data quality monitoring | Automated checks for completeness and accuracy |
| Initial data analysis report | Patterns, distributions, anomalies in collected data |
| Feature engineering plan | Identified features for ML model based on data analysis |

## 4. Phase 3: Initial ML Model

### 4.1 Objective
Train and deploy the **first ML model** once sufficient historical data is available (≥ 100 completed orders).

### 4.2 Algorithm Selection

| Algorithm | Pros | Cons | Use When |
|-----------|------|------|----------|
| **Linear Regression** | Simple, interpretable, fast | Assumes linear relationships | Baseline model, explainability needed |
| **Random Forest** | Handles non-linearity, robust | Less interpretable, larger file | 100-200 orders, moderate complexity |
| **Gradient Boosting** | Higher accuracy, handles complex patterns | Requires careful tuning | 200+ orders |

**Recommended Starting Algorithm:** Random Forest (good balance of accuracy and robustness with small datasets)

### 4.3 Feature Set (Phase 3)

| Feature | Type | Source |
|---------|------|--------|
| `ingredient_cost` | Numeric | Recipe × Price calculation |
| `quantity` | Numeric | Request input |
| `ingredient_count` | Numeric | Recipe data |
| `recipe_complexity_score` | Numeric | Derived from ingredient count + category |
| `category_encoded` | Categorical (one-hot) | Recipe metadata |
| `month` | Numeric | Event date |
| `is_wedding_season` | Binary | Derived from month |
| `is_festival_season` | Binary | Derived from month |
| `guest_count` | Numeric | Request input (if available) |

### 4.4 Training Pipeline

```
┌──────────────────────────────────────────────────────────────┐
│                  PHASE 3 TRAINING PIPELINE                   │
│                                                              │
│  1. Data Extraction                                          │
│     └─▶ Pull completed orders with actual costs              │
│     └─▶ Join with prediction logs                            │
│     └─▶ Join with recipe + price data                        │
│                                                              │
│  2. Feature Engineering                                      │
│     └─▶ Calculate all features from raw data                 │
│     └─▶ Handle missing values (imputation or exclusion)      │
│     └─▶ Normalize numeric features if needed                 │
│                                                              │
│  3. Data Split                                               │
│     └─▶ 70% Training / 15% Validation / 15% Test            │
│     └─▶ Stratified by menu category                          │
│     └─▶ No temporal leakage (train on older, test on newer)  │
│                                                              │
│  4. Model Training                                           │
│     └─▶ Train Linear Regression (baseline)                   │
│     └─▶ Train Random Forest (primary)                        │
│     └─▶ Hyperparameter tuning (GridSearchCV)                 │
│     └─▶ Cross-validation (5-fold)                            │
│                                                              │
│  5. Evaluation                                               │
│     └─▶ MAPE on test set (target: < 15%)                    │
│     └─▶ MAE, RMSE, R² on test set                           │
│     └─▶ Residual analysis (no systematic patterns)           │
│     └─▶ Feature importance analysis                          │
│     └─▶ Edge case testing (new items, extreme quantities)    │
│                                                              │
│  6. Deployment Decision                                      │
│     └─▶ If MAPE < 15%: Deploy in A/B mode alongside rules   │
│     └─▶ If MAPE ≥ 15%: Iterate on features/algorithm        │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 4.5 A/B Testing Plan (Phase 3)
- Deploy ML model in **shadow mode** first (1 week)
- Then 10% traffic to ML model (1 week)
- Evaluate: if ML MAPE ≤ rule-based error → scale to 50%
- If 50% performs well (1 week) → scale to 100%
- Keep rule-based as permanent fallback

### 4.6 Phase 3 Success Criteria

- [ ] ML model trained with MAPE < 15% on test set
- [ ] Cross-validation scores consistent across folds
- [ ] No data leakage detected
- [ ] Feature importance aligns with business intuition
- [ ] A/B testing shows ML model ≥ rule-based performance
- [ ] Latency < 500ms (P99) with model inference

## 5. Phase 4: Advanced ML Model

### 5.1 Objective
Train a **production-grade ML model** with expanded features and sophisticated algorithms, achieving MAPE < 10%.

### 5.2 Expanded Feature Set

| Feature | Type | New in Phase 4 |
|---------|------|----------------|
| All Phase 3 features | Various | No |
| `price_volatility_30d` | Numeric | ✅ Historical price variance |
| `demand_index` | Numeric | ✅ Order frequency signal |
| `day_of_week` | Numeric | ✅ Weekday vs weekend |
| `days_until_event` | Numeric | ✅ Lead time indicator |
| `historical_avg_cost` | Numeric | ✅ Same item historical average |
| `customer_segment` | Categorical | ✅ Repeat vs new customer |
| `event_type` | Categorical | ✅ Wedding/corporate/birthday |
| `location` | Categorical | ✅ Regional pricing |

### 5.3 Advanced Algorithms

| Algorithm | Expected MAPE | Training Time | Notes |
|-----------|---------------|---------------|-------|
| **XGBoost** | < 10% | Minutes | Primary candidate |
| **LightGBM** | < 10% | Minutes | Alternative, faster training |
| **Stacking Ensemble** | < 8% | Minutes | Combine multiple models |

### 5.4 Phase 4 Success Criteria

- [ ] MAPE consistently < 10% over 30-day rolling window
- [ ] Deployed as primary prediction method (100% traffic)
- [ ] Automated monitoring and alerting operational
- [ ] Automated retraining pipeline configured
- [ ] Model drift detection active
- [ ] Rollback plan tested and documented

## 7. Risk Assessment

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|------------|
| Insufficient order data by Month 4 | Medium | High (delays Phase 3) | Extend data collection period; consider synthetic augmentation |
| Recipe data gaps persist | Medium | Medium | Prioritize filling top 20 most-ordered recipes |
| Ingredient price volatility | Low | Medium | Regular price updates; staleness alerts |
| ML model underperforms rule-based | Low | Low | Keep rule-based as permanent fallback |
| Team bandwidth constraints | Medium | Medium | Prioritize Phase 1 completion; Phase 2 is passive collection |
