## 2. Feature Descriptions & Rationale

### 2.1 Base Features (Available Now)

| # | Feature | Type | Range | Rationale | Expected Importance |
|---|---------|------|-------|-----------|-------------------|
| 1 | `base_ingredient_cost` | Numeric | ₹0 — ₹50,000+ | Primary cost driver — ingredients are 60-70% of total cost | **Very High** |
| 2 | `quantity` | Numeric | 1 — 10,000 | Scales cost linearly; affects bulk discounts | **High** |
| 3 | `recipe_complexity` | Numeric | 1 — 27 | More ingredients = generally higher cost and labor | **Medium** |
| 4 | `menu_category` | Categorical | 7 categories | Non-veg costs more than veg; desserts differ from mains | **Medium** |
| 5 | `has_perishable` | Binary | 0/1 | Perishable ingredients (meat, dairy) cost more, higher wastage | **Low-Medium** |

### 2.2 Time-Based Features (Require Order History)

| # | Feature | Type | Range | Rationale | Expected Importance |
|---|---------|------|-------|-----------|-------------------|
| 6 | `month` | Categorical | 1-12 | Captures seasonal ingredient price changes and demand | **Medium** |
| 7 | `day_of_week` | Categorical | 0-6 | Weekend events may have premium pricing | **Low** |
| 8 | `is_weekend` | Binary | 0/1 | Weekend = higher demand, potentially higher costs | **Low-Medium** |
| 9 | `is_wedding_season` | Binary | 0/1 | Oct-Feb peak → 15% seasonal premium | **Medium** |
| 10 | `is_festival_season` | Binary | 0/1 | Mar-Apr festivals → 10% premium | **Low-Medium** |

### 2.3 Demand Features (Require Order History)

| # | Feature | Type | Range | Rationale | Expected Importance |
|---|---------|------|-------|-----------|-------------------|
| 11 | `historical_demand` | Numeric | 0 — 100+ | High-demand items may have bulk pricing advantages | **Medium** |
| 12 | `days_until_event` | Numeric | 0 — 365 | Last-minute orders may cost more (rush pricing) | **Low-Medium** |

### 2.4 Price Features (Require Price History)

| # | Feature | Type | Range | Rationale | Expected Importance |
|---|---------|------|-------|-----------|-------------------|
| 13 | `price_volatility` | Numeric | 0.0 — 1.0+ | Volatile ingredients → less predictable costs | **Medium** |
| 14 | `recent_price_trend` | Numeric | -1.0 — 1.0 | Rising prices → higher predicted cost | **Medium** |

### 2.5 Contextual Features (From Order Request)

| # | Feature | Type | Range | Rationale | Expected Importance |
|---|---------|------|-------|-----------|-------------------|
| 15 | `guest_count` | Numeric | 1 — 5000 | Larger events may have different cost dynamics | **Low-Medium** |
| 16 | `event_type` | Categorical | 4 types | Wedding vs Corporate vs Birthday vs Other pricing | **Medium** |

## 4. Data Leakage Prevention Checklist

| # | Check | Description | Status |
|---|-------|-------------|--------|
| 1 | **No target leakage** | `actual_cost` or `total_cost` is NEVER used as a feature | ✅ Verified |
| 2 | **No future data** | All features are computed from data available BEFORE prediction time | ✅ Verified |
| 3 | **Temporal integrity** | Training data is from before test data (no time-travel) | ✅ Design enforced |
| 4 | **Independent features** | `base_ingredient_cost` is computed from recipe × price, not from actual order cost | ✅ Verified |
| 5 | **No test contamination** | Scaling parameters (mean, std) are fitted on training data ONLY | ✅ Pipeline enforces |
| 6 | **Prediction-time availability** | All 16 features can be computed at the moment `/predict-cost` is called | ✅ Verified |
| 7 | **No label encoding leakage** | Category encoders fitted on training data only, unknown categories handled | ✅ Pipeline enforces |

### Potential Leakage Risks to Watch:

| Risk | Scenario | Prevention |
|------|----------|-----------|
| Using actual cost as feature | Accidentally including `actual_total_cost` in training features | Never include columns starting with `actual_` |
| Future price data | Using ingredient prices from AFTER the order date | Use price snapshot at prediction time |
| Aggregate leakage | `historical_demand` includes demand from future orders | Calculate demand only from orders BEFORE current date |
| Cross-validation leakage | Scaling on full dataset before splitting | Always scale within cross-validation folds |

## 5. Feature Engineering Principles

1. **Prediction-time availability** — Every feature must be computable at the moment the API receives a prediction request
2. **Interpretability** — Every feature should have a clear business explanation for why it affects cost
3. **Robustness** — Handle missing values gracefully (imputation, defaults)
4. **No leakage** — Never use information from after the prediction time
5. **Efficiency** — Features should be fast to compute (< 10ms per prediction)
6. **Scalability** — Feature pipeline should work for 1 prediction or 10,000 batch predictions

## 6. Feature Interaction Expectations

| Interaction | Expected Effect |
|-------------|----------------|
| `quantity × base_ingredient_cost` | Linear scaling — cost grows proportionally |
| `is_wedding_season × quantity` | Large wedding orders during peak → highest cost |
| `recipe_complexity × has_perishable` | Complex recipes with perishable items → higher wastage costs |
| `event_type × guest_count` | Corporate events with 500+ guests → potential volume discount |
| `price_volatility × month` | Volatile ingredients during monsoon → unpredictable costs |
