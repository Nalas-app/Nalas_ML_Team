# Documentation Maintenance & Governance Schedule

**Maintainer:** Adwaitha (ML Lead)  
**Last Updated:** 2026-03-28

This document outlines the strict protocol and schedule for reviewing, updating, and expiring technical documentation across the Nalas ML Costing Engine stack.

---

## 1. Living Documents (Update Frequently)

These files are expected to be updated *simultaneously* with code commits.

| Document | Trigger | Responsibility |
|----------|---------|----------------|
| **`6.4_model_card.md`** | Following *any* successful automated or manual retraining deployment. | `phase6_retrain.py` (future auto-write) / Adwaitha |
| **`7.1_api_integration_guide.md`** | Any change to the request/response JSON schema in `main.py`. | Vasanth / Adwaitha |
| **`feature_pipeline.py` docstrings** | Any new feature added to the extraction matrix list. | Adwaitha |
| **`business_rules.json`** | Multiplier changes (profit%, labor%). | Head Chef / Ops Lead |

---

## 2. Version-Specific Artifacts (Create on Release)

Do not overwrite historical performance metrics. Maintain traceability.

| Document | Trigger | Output Location |
|----------|---------|-----------------|
| **`metadata.json`** | Generated upon each release in `model_registry.py`. | `model/vX.X/metadata.json` |
| **`retrain_run.json`** | Generated by `phase6_retrain.py` after tuning runs. | `logs/retraining/retrain_YYYYMMDD.json` |
| **`feature_importance.csv`** | Generated by `FeatureAnalyzer` during phase 4/8 tuning checks. | `data/model_evaluation/` |

---

## 3. Operational Playbooks (Update as Discovered)

Operational guides grow through incident discovery. Post-mortems drive documentation updates.

* **`8.2_dashboard_and_alerting_playbook.md`:** Must be updated immediately after an incident if the SLA or investigation action changed.
* **Troubleshooting Logs (`app.log`):** Must rotate automatically ensuring no single file exceeds 500MB on the server.
* **`test_requests.json`:** If a new valid business case edge boundary is discovered, it must be added to the test battery suite JSON.

---

## 4. Recurring Review Schedule

| Frequency | Action Required |
|-----------|-----------------|
| **Weekly** | Ops Lead briefly reviews the `feedback` tables for MAPE drift (via `phase8_feedback_loop.py` logs) to confirm if manual intervention is required over the weekend. |
| **Monthly** | Adwaitha reviews the `Model Card` limitations section. Any false-positive flags or outdated limitation statements are removed. |
| **Quarterly** | Full architecture document review (Phases 1-8). Check `requirements.txt` for severely deprecated SDKs (e.g. Scikit-learn CVEs). Ensure PagerDuty escalation policies listed in the Alerting Playbook belong to active employees. |

---

## 5. Version Control Best Practices

* Documentation commits should run in parallel with the code branch they cover.
* Never merge to `main` without updating the associated API examples.
* Major version updates (`v2.0.0` etc.) should bundle all Phase 7 handoff artifacts into a new `ML_Costing_Release_v2.0.zip` automatically to ensure Vasanth integrates the exact documentation tied to the software execution state.
