# ML Costing Model — Improvement Roadmap

**Maintainer:** Adwaitha (ML Lead)  
**Last Updated:** 2026-03-28

This roadmap highlights the immediate, medium, and long-term MLOps and mathematical improvements planned for the Nalas Catering Cost prediction engine post-v1.0.0 handoff.

---

## 📅 Short-Term Improvements (1-3 Months)

**Focus: Feature Enrichment & Robustness**
The v1.0.0 model works effectively but heavily depends on core ingredient ratios. We need more context.

1. **Rich Customer Features:**
   * Include customer segmentation data (`corporate_vip`, `retail`, `recurring`).
   * Introduce historical demand volume mapped back to exactly 1 year ago (Month + Same Day).
2. **Ingredient Price Trending:**
   * Currently, the DB pulls "latest" static prices. We need to scrape moving averages over 7/14/30 days and embed volatility indices into the `FeaturePipeline`.
3. **Advanced Feature Engineering:**
   * **Polynomial interactions:** Multipliers crossing `guestCount * recipe_complexity`.
   * **Time-based aggregations:** E.g., moving window sums of the last 14 days of ordering to capture sudden wedding-season spikes properly.
4. **Hyperparameter Grids:**
   * Expand `GridSearchCV` boundaries beyond our Phase 5 settings: test deeper trees (`max_depth: 15`) with higher shrinkage rates (`learning_rate: 0.01`).

---

## 📅 Medium-Term Improvements (3-6 Months)

**Focus: Ensemble Logic & Interpretability**
Once we hit >5,000 finalized real-world feedback loops, we can shift away from standard Scikit-Learn tree models.

1. **Ensemble Modeling Architecture:**
   * Combine Gradient Boosting with Random Forests and an underlying Linear Regression layer via a Soft Voting mechanism where the weights correspond directly to `confidence_score` calculations.
2. **Time-Series Sub-Models:**
   * Instead of predicting cost linearly relative to static prices, implement a secondary **Prophet or LSTM** model to explicitly forecast market commodity prices for base ingredients (like Onion, Mutton) so quotes 4 months out incorporate inflation auto-magically.
3. **Model Interpretability (XAI):**
   * Integrate **SHapley Additive exPlanations (SHAP)** into the FastAPI response so the Chef can see exactly *why* a quote is Rs.10,000 (e.g. `+Rs.400 because wedding season`, `+Rs.1000 because market Mutton price is volatile this week`).

---

## 📅 Long-Term Improvements (6-12 Months)

**Focus: Dynamic Output & Autonomous Pricing**

1. **Personalization Models:**
   * Build per-category specific routing models (e.g., A completely distinct Neural Network that ONLY predicts Non-Veg BBQ costs, rather than 1 monolithic engine).
2. **Dynamic Pricing Integration:**
   * Right now we predict `Cost` using `profit_margin = 10%`. We should build a true `PriceOptimizer` agent that watches competitor catering menus and tracks daily demand elasticity to suggest changing profit margins from `[5% - 25%]` dynamically.
3. **Multi-Objective Optimization:**
   * Train models on a combined loss function that minimizes `Cost Error (MAPE)` while concurrently maximizing `Conversion Rate` and `Net Margin`.

---

## 🧪 Research Backlog Topics
* **Causal Inference Deep Dive:** Identifying *why* certain recipes consistently under-cost on weekends. (Is it ingredient spillage and wastage during rush hours, rather than baseline recipe mismatches?)
* **Anomaly Detection Pipelines:** Pre-processing anomaly detection specifically designed to flag typographical errors in Menu Item DB imports (e.g. `1 Kg Saffron` vs `1 gram Saffron`) before the ML model trains on ruined data.
