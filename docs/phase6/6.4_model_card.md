---

## 2. Intended Use

### Primary Use
- Generate cost quotations for customer catering orders.
- Provide itemized cost breakdowns (ingredients, labor, overhead, profit).

### Secondary Use
- Cost estimation for menu planning and pricing decisions.
- Ingredient budget forecasting for procurement.

### Out of Scope
- Custom menu items not in the database
- Raw material market price prediction
- Multi-venue or multi-day event pricing
- Competitor pricing analysis

## 7. Known Limitations

1. **Synthetic training data** — v1.0.0 trains on simulated data, not real orders. MAPE will improve when retrained on production data.
2. **Incomplete recipes** — Only 26 of 34 menu items (76.5%) have sufficient recipe data for prediction.
3. **Static prices** — Ingredient prices are loaded from CSV at startup. No real-time price feeds.
4. **Unit inconsistencies** — Some recipes have KG vs gram mismatches causing anomalous costs (Mutton Varuval, Pongal).
5. **No custom items** — Cannot predict cost for menu items not in the database.
6. **Seasonality is approximated** — Wedding/festival season adjustments are rule-based (15%, 10%), not learned from data.
7. **No demand-based pricing** — Historical demand feature is empty until orders are collected.
8. **Price history features empty** — Volatility and trend features require 3+ months of price snapshots.

## 8. Bias & Fairness Analysis

### Systematic Check

| Dimension | Bias Check | Result |
|-----------|-----------|--------|
| By Menu Category | Non-veg predictions higher than veg? | ✅ Expected — non-veg ingredients cost more |
| By Order Size | Large orders under-predicted? | ⚠️ Slight under-prediction for qty > 500 (fewer training examples) |
| By Season | Wedding season over-predicted? | ✅ 15% premium is consistent and documented |
| By Event Type | Any event type systematically different? | ✅ No — event type has < 0.3% feature importance |

### Identified Biases
1. **Large order bias** — Predictions for qty > 500 have higher error due to sparse training data in that range. Mitigation: Collect more large-order data.
2. **Item-specific bias** — Items with fewer matched ingredients have lower confidence. Mitigation: Improve ingredient name matching.

### Fairness Statement
The model does not use any protected attributes (customer demographics, location, etc.) as features. Cost predictions are based solely on menu item properties, quantity, and temporal factors.

## 9. Maintenance & Retraining

### Schedule

| Trigger | Condition | Action |
|---------|-----------|--------|
| Scheduled | Monthly (1st of month) | Retrain if sufficient new data |
| Performance | 7-day rolling MAPE > 15% | Trigger immediate retrain |
| Data Volume | 200+ new completed orders | Retrain with expanded dataset |
| Manual | ML team decision | On-demand retraining |

### Retraining Process
1. Collect new order data from prediction logs
2. Re-extract features using saved pipeline
3. Retrain Gradient Boosting with GridSearchCV
4. Evaluate on recent orders (last month as test set)
5. Compare with current production model
6. Deploy if MAPE improves or is within 2% of current

### Rollback Criteria
- New model MAPE > old model MAPE + 2%
- New model R² < 0.80
- Prediction latency > 500ms
- Any critical error in production within 24 hours of deployment

### Rollback Command
```bash
python scripts/phase6_retrain.py --version v1.0.0 --force
```

## 10. Ethical Considerations

1. **Transparency** — All predictions include confidence scores. Low-confidence predictions (< 0.60) are flagged for manual review.
2. **Availability** — Rule-based fallback ensures predictions are always available, even if the ML model fails.
3. **No discrimination** — Model uses only item-level and temporal features. No customer-level features are used.
4. **Auditability** — Every prediction is logged as JSONL with full request/response data.
5. **Human oversight** — Final pricing decisions remain with the business team. ML predictions are recommendations, not binding quotes.
