## 1. Retraining Pipeline Script

The fully automated retraining pipeline is implemented in **`scripts/phase6_retrain.py`**.

This script handles:
1. **Trigger Evaluation:** Checks if scheduled, performance, or data criteria are met.
2. **Data Ingestion:** Merges historical rule-based data with new real-time order data logged by the `PredictionLogger`.
3. **Model Training:** Reruns the `FeaturePipeline` and trains a new `GradientBoostingRegressor` using `GridSearchCV` to tune hyperparameters automatically.
4. **Evaluation:** Computes the MAPE, R², RMSE, and MAE against the test set.
5. **Deployment:** Passes the metrics to the `ModelRegistry` module to decide if the model mathematically improves upon the current active version.
6. **Logging:** Dumps a detailed `retrain_*.json` report into `logs/retraining/`.

## 2. Retraining Schedule Configuration

To automate the retraining pipeline, it should be added to the server's scheduling service (e.g., `cron` on Linux/macOS or Task Scheduler on Windows). 

Even if scheduled to run daily, the script internally checks the triggers and **will abort automatically** if conditions (like 200 new orders or 30 days passing) are not met.

### Recommended `crontab` Configuration (Linux/macOS)
```bash
# Run the retraining evaluation script every day at 2:00 AM server time
0 2 * * * cd /path/to/nalas_ml_costing/nalas_app/ML_costing && python -X utf8 scripts/phase6_retrain.py >> logs/retraining/cron.log 2>&1
```

### Windows Task Scheduler (PowerShell Equivalent)
```powershell
# Create a daily scheduled task to run the retraining script
$action = New-ScheduledTaskAction -Execute "python" -Argument "-X utf8 scripts/phase6_retrain.py" -WorkingDirectory "C:\Users\sathe\OneDrive\Documents\nalas_ml_costing\nalas_app\ML_costing"
$trigger = New-ScheduledTaskTrigger -Daily -At 2:00AM
Register-ScheduledTask -Action $action -Trigger $trigger -TaskName "Nalas_ML_Retraining" -Description "Daily ML Model Evaluation"
```

## 3. Deployment Decision Logic

The deployment decision logic is physically implemented in **`ml_service/model_registry.py`** under the `should_deploy()` method.

When a new model completes training, it evaluates the test **MAPE (Mean Absolute Percentage Error)** against the currently active production model. The rules are strict:

| Condition | Action | Rationale |
|-----------|--------|-----------|
| **New MAPE < Current MAPE** | **✅ Deploy** | The new model is mathematically more accurate. Promoted to `active`. |
| **New MAPE > Current MAPE + 2.0%** | **❌ Reject** | Severe degradation. The new dataset likely contains massive outliers or data drift. Kept as inactive for review. |
| **Current MAPE ≤ New MAPE ≤ Current MAPE + 2.0%** | **✅ Deploy** | Similar performance. Promoted to take advantage of learning the latest recent data patterns (menu changes, recent seasonal shifts). |

## 4. Rollback Procedures

If a newly deployed model behaves unexpectedly in production, the `ModelRegistry` makes reverting to the previous model instantaneous (sub-second) and does not require restarting the API layer.

### How to Rollback
Run the retraining script with the `--force` flag targeting the exact previous version string.

```bash
# Example: Production just updated to v1.1.0 and is predicting poorly. 
# Revert back to v1.0.0 immediately:

python scripts/phase6_retrain.py --version v1.0.0 --force
```

### What happens during a rollback?
1. The `ModelRegistry` redirects the `./model/active/` symlink/folder back to the `./model/v1.0.0/` folder directory.
2. It updates `model/registry.json` to mark `v1.0.0` as `active` again.
3. Next time the ` प्रोडक्शनPredictor` is called or hot-reloaded, it will seamlessly begin serving predictions using the older `model.pkl` and `pipeline.pkl` files without dropping API requests.

## 5. Monitoring Retraining Post-Deployment

Whenever a retrain successfully deploys a new model, it generates a full report in `logs/retraining/retrain_{timestamp}.json`. 

You should periodically check this directory to review the JSON logs, which will indicate exact reasons *why* a model was approved or rejected:

```json
{
  "run_id": "20260328_123000",
  "version": "v1.1.0",
  "training_samples": 820,
  "metrics": {
    "mape": 9.42,
    "rmse": 230491.12,
    "r2": 0.981
  },
  "decision": {
    "deploy": true,
    "reason": "Improvement: MAPE 14.49% → 9.42%"
  }
}
```
