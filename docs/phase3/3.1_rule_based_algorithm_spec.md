## 1. Algorithm Overview

The rule-based cost engine calculates menu item costs deterministically using:
- **Recipe data:** Ingredient quantities per batch from NRC Excel files
- **Ingredient prices:** 129-item price master from NRC Kitchen file
- **Business multipliers:** Configurable percentages for labor, overhead, wastage, profit

### Purpose
Provide immediate, deterministic cost predictions while collecting real order data for future ML model training.

---

## 2. Algorithm Steps

```
INPUT: menu_item_id, quantity, event_date (optional), guest_count (optional)

STEP 1: VALIDATE INPUTS
        - quantity > 0? If not → ML_001 error
        - menu_item_id exists? If not → ML_002 error

STEP 2: FETCH RECIPE
        - Look up recipe for menu item from structured data
        - If no recipe found → ML_003 error

STEP 3: CALCULATE INGREDIENT COSTS
        For each ingredient in recipe:
            - Get ingredient price from master
            - ingredient_cost = quantity_per_base × price × wastage_factor(1.05)
            - Skip if: UNMATCHED, no quantity, or no price
        total_ingredient_cost = SUM(all ingredient costs) × quantity

STEP 4: APPLY BUSINESS MULTIPLIERS
        labor_cost     = total_ingredient_cost × 0.15
        overhead_cost  = total_ingredient_cost × 0.10
        wastage_cost   = total_ingredient_cost × 0.05
        subtotal       = total_ingredient_cost + labor + overhead + wastage

STEP 5: APPLY SEASONAL ADJUSTMENT (if event_date provided)
        Wedding season (Oct-Feb): +15%
        Festival season (Mar-Apr): +10%
        Off season (May-Sep): +0%

STEP 6: APPLY PROFIT MARGIN
        profit_margin = subtotal × 0.10
        final_cost = subtotal + profit_margin

STEP 7: APPLY BULK DISCOUNT
        1-100 servings:    0% discount
        101-300 servings:  5% discount
        301-500 servings:  8% discount
        501-1000 servings: 10% discount
        1001+ servings:    12% discount

STEP 8: ENFORCE MINIMUM COST
        final_cost = MAX(final_cost, min_cost_per_serving × quantity)

STEP 9: CALCULATE CONFIDENCE
        Based on ingredient match rate + data completeness
        Rule-based cap: 85% (ML can go higher in later phases)

STEP 10: RETURN RESPONSE
        {ingredientCost, laborCost, overheadCost, totalCost,
         confidence, modelVersion, method: "rule_based"}

OUTPUT: Cost breakdown matching API contract schema
```

## 3. Business Rules Configuration

All multipliers are stored in `ml_service/config/business_rules.json` and can be changed without code changes.

| Parameter | Value | Editable |
|-----------|-------|----------|
| Labor % | 15% of ingredient cost | Yes (config) |
| Overhead % | 10% of ingredient cost | Yes (config) |
| Wastage % | 5% of ingredient cost | Yes (config) |
| Profit % | 10% of subtotal | Yes (config) |
| Wastage Factor | 1.05x per ingredient | Yes (config) |
| Model Version | v0.1.0 | Yes (config) |
| Cache TTL | 3600 seconds | Yes (config) |

## 4. Edge Case Handling

| Edge Case | Handling | Error Code |
|-----------|---------|------------|
| Menu item not found | Return error | ML_002 |
| Recipe not found | Return error | ML_003 |
| No priced ingredients | Return error | ML_004 |
| Missing ingredient price | Skip ingredient, reduce confidence | N/A (flag) |
| Quantity = 0 | Return validation error | ML_001 |
| Quantity < 0 | Return validation error | ML_001 |
| No event date | Skip seasonal adjustment | N/A |
| Invalid event date format | Skip seasonal adjustment | N/A |
| Cost below minimum | Enforce minimum cost per serving | N/A (flag) |
| Partial recipe data | Calculate with available data, lower confidence | N/A (flag) |

---

## 5. Confidence Scoring

| Factor | Weight | Logic |
|--------|--------|-------|
| Ingredient match rate | Primary | % of recipe ingredients matched to price master |
| Matched count bonus | +5-10% | 5+ matched = +5%, 10+ matched = +10% |
| Unmatched penalty | -30% | If unmatched > matched count |
| Rule-based cap | 85% max | Rule-based method cannot exceed 85% confidence |
| Floor | 10% min | Confidence never below 10% |

---

## 6. Limitations

1. **No historical learning** — costs are deterministic, not trained on past data
2. **No demand-based pricing** — does not adjust for high-demand periods (except seasonal)
3. **Partial data** — 38.4% of recipe-ingredient references are unmatched
4. **No per-serving granularity** — recipes are per batch, not per serving
5. **Static prices** — prices don't update automatically (manual CSV update needed)
